## Design Context

### Users

Zoosite has two main user groups for this work. Public visitors are business owners evaluating site, blog, and digital presence services from Zoosite. Protected blog users are Zoosite operators, editors, reviewers, publishers, media managers, moderators, analysts, and administrators who understand the Zoolanding generic-component system and need to create SEO-oriented articles without exposing implementation details to clients.

### Use Cases

- Public visitors browse articles, category pages, and article detail pages for practical guidance and SEO discovery.
- Blog operators create, edit, preview, validate, review, approve, publish, schedule, unpublish, archive, restore, and analyze articles.
- Administrators configure taxonomy, moderation defaults, media metadata, and role-specific access without writing draft code for every routine operation.
- Advanced users may use richer component and JSON controls, but those controls must be gated and safe by draft configuration.

### Brand Personality

Practical, precise, and confident. The public site should feel clear and commercially useful, not like an internal tool. The protected blog admin should feel dense enough for repeated work, but still readable and calm.

### Aesthetic Direction

Use the existing Zoosite visual language and shared site chrome as the baseline: light UI, restrained surfaces, strong readable typography, green/teal action accents, clear spacing, and robust mobile touch targets. Avoid AI-looking decoration, unexplained acronyms, internal implementation labels, nested cards, tiny action links, and one-off draft-specific Angular screens.

### Design Principles

- Drafts compose experiences with generic components; app code adds primitives only when the existing generic layer cannot express a reusable need.
- Public copy must be visitor-facing. Admin copy must explain the job to be done, not expose backend, BFF, content-hub, bundle, package, bucket, table, or raw variable names.
- Every action that can mutate content needs visible loading, disabled, success, and safe error states.
- Mobile touch targets must be usable, especially table actions, auth controls, pagination, dropdowns, and editor toolbar buttons.
- SEO surfaces must be content-aware and host-aware: published public articles only, correct canonical/robots/search/sitemap/feed behavior, and no shared-preview query dependency in production.
